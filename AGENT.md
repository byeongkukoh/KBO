# AGENT.md

이 문서는 KBO Record 저장소에서 사람이든 AI든 동일한 기준으로 작업하기 위한 운영 규칙 문서다.

## 1. Project Roles

### Product Planning

- 서비스 범위, 화면 구조, 데이터 우선순위를 `docs/product/` 기준으로 관리한다.
- 새로운 기능을 구현하기 전에 MVP 포함 여부와 제외 여부를 먼저 문서에 반영한다.

### Frontend

- 위치: `apps/web`
- 역할: 사용자에게 선수, 팀, 경기 기록 화면을 제공한다.
- 책임: 탐색 경험, 필터링, 상세 화면 구성, 데이터 최신성 표시

### Backend API

- 위치: `apps/api`
- 역할: 프론트엔드가 사용하는 기록 조회 API를 제공한다.
- 책임: 시즌 기록, 선수 기록, 팀 기록, 경기 기록, 파생 지표 응답

### Scraping / Data Pipeline

- 현재는 별도 앱이 없지만 향후 `apps/scraper` 또는 `apps/worker` 로 분리하는 것을 기본 방향으로 삼는다.
- 역할: KBO 공식 홈페이지를 원천 데이터 소스로 사용해 데이터를 수집하고 정제한다.
- 기본 스크래핑 라이브러리 후보는 Python `SCRAPLING` 으로 둔다.

### Documentation

- 위치: `docs/`
- 역할: 기획, 데이터 모델, 운영 규칙, 아키텍처 결정을 기록한다.
- 구현 전에 먼저 문서가 바뀌어야 하는지 확인하고, 변경이 필요하면 코드와 함께 업데이트한다.

## 2. Working Principles

- 모든 작업은 현재 문서와 저장소 구조를 먼저 확인한 뒤 시작한다.
- 추측으로 도메인 규칙을 만들지 않는다. 불확실한 사항은 문서에 가정으로 명시한다.
- 선수, 팀, 경기 데이터는 각각 다른 조회 단위를 가지므로 같은 구조로 억지 통합하지 않는다.
- 경기 상세 데이터는 가능한 한 raw 데이터에 가깝게 보존하고, 파생 지표는 별도 계산 계층으로 다룬다.
- 문서 구조와 코드 구조가 어긋나지 않도록 유지한다.

## 3. Documentation Conventions

- 서비스 목적과 범위는 `docs/product/` 에 기록한다.
- 데이터 엔티티와 통계 항목은 `docs/data/` 에 기록한다.
- 스크래핑, 배치, 실패 처리, 운영 흐름은 `docs/operations/` 에 기록한다.
- 시스템 경계와 프로젝트 구조는 `docs/architecture/` 에 기록한다.
- 새로운 문서는 한 주제만 다루고, 파일명은 소문자 kebab-case를 사용한다.
- 문서에는 확정 사항과 향후 검토 사항을 구분해서 적는다.

## 4. Git Conventions

### Branching

- 기본 브랜치는 `main` 이다.
- 기능 작업은 필요 시 `feature/<topic>` 형식을 사용한다.
- 문서 작업은 필요 시 `docs/<topic>` 형식을 사용한다.
- 버그 수정은 필요 시 `fix/<topic>` 형식을 사용한다.

### Commit Timing

- 하나의 논리적 작업 단위가 끝났을 때 커밋한다.
- 스캐폴드, 문서 구조 생성, 기능 추가, 리팩터링, 버그 수정은 가능한 한 분리 커밋한다.
- 실패한 중간 상태, 생성 산출물, 검증되지 않은 결과는 커밋하지 않는다.
- 문서가 설계를 바꾸는 경우 문서 변경을 구현보다 먼저 또는 같은 커밋에 포함한다.

### Commit Message Rules

- 커밋 메시지는 명령형 현재 시제로 작성한다.
- 첫 줄은 짧고 구체적으로 작성한다.
- 권장 형식은 다음 중 하나를 따른다.
  - `Add ...`
  - `Update ...`
  - `Fix ...`
  - `Refine ...`
  - `Document ...`

예시:

- `Add FastAPI health check endpoint`
- `Document player and game data scope`
- `Refine frontend TypeScript build config`

### Commit Content Rules

- `node_modules`, 빌드 산출물, 테스트 캐시, 임시 파일은 커밋하지 않는다.
- 자동 생성 파일이 필요하지 않다면 `.gitignore` 로 제외한다.
- 하나의 커밋에는 하나의 이유가 드러나야 한다.
- 문서 전용 커밋과 코드 전용 커밋을 무조건 분리할 필요는 없지만, 같은 목적을 설명할 수 있어야 한다.

## 5. Verification Rules

- Python 코드를 바꾸면 최소한 관련 테스트 또는 실행 검증을 수행한다.
- 프론트엔드 코드를 바꾸면 최소한 빌드 검증을 수행한다.
- 문서를 바꾸면 관련 문서 간 충돌이 없는지 다시 읽는다.
- 스크래핑 관련 설계는 원천 소스, 수집 주기, 실패 처리, 중복 방지 전략이 함께 문서화되어야 한다.

## 6. Planning Rules For This Project

- 선수 페이지는 선수 커리어 기준의 시즌별 기록을 중심으로 설계한다.
- 팀 페이지는 팀의 시즌별 누적 기록을 중심으로 설계한다.
- 경기 페이지는 가능한 한 raw한 경기 흐름과 이벤트 데이터를 보여주는 방향으로 설계한다.
- 고급 지표는 원천 기록과 리그 기준 데이터가 준비된 이후 계산한다.
- PostgreSQL은 주 저장소로 사용하되, 세부 스키마는 데이터 모델 문서에서 확정한다.
- 시즌 진행 중에는 하루 1회 데이터 스크래핑 및 업데이트를 기본 정책으로 한다.

## 7. Change Checklist

작업 전 확인:

- 관련 문서가 이미 있는가?
- 이번 변경이 문서, 코드, 둘 다 수정해야 하는가?
- 현재 작업이 하나의 논리적 커밋으로 설명 가능한가?

작업 후 확인:

- 관련 문서가 최신 상태인가?
- 불필요한 생성 파일이 포함되지 않았는가?
- 검증 결과가 있는가?
- 다음 작업자가 바로 이해할 수 있는 상태인가?
